# 台股主動式 ETF 持股追蹤 Dashboard — 設計文件

日期：2026-07-04
狀態：已與使用者逐節確認

## 1. 目標

追蹤台股全部主動式**股票型** ETF（代號結尾 `A`，目前 27 檔），每日更新：

- 各 ETF 增持 / 減持 / 新進 / 出清了哪些股票
- 各 ETF 當前完整持股比例
- 各 ETF 績效（月/季/年報酬）與兩種勝率（擇時、選股）
- 跨 ETF 彙總視角（個股反查、集體動向、人氣持股）

產出為公開網頁 Dashboard，總營運成本 $0/月。

**海外持股範圍限定（v1）**：部分 A 結尾 ETF 持有海外股票（如 00402A、00989A、00997A 等美股型）。其持股快照與異動照常追蹤，ETF 層級指標（報酬、擇時勝率）照常計算；但**個股層級指標（選股勝率、個股反查）僅涵蓋台股持倉**——海外個股行情整合列為後續擴充。`etf` 表以 `universe`（tw/global）欄位區分，代號有效性驗證僅適用台股持倉。

## 2. 資料來源查證結論

| 來源 | 查證結果 |
|---|---|
| FinMind | 完整 dataset 清單中**無** ETF 持股明細 → 只用於股價/指數/股票基本資料 |
| TWSE OpenAPI | 全端點掃描，**無** PCF 或 ETF 持股資料 |
| 投信投顧公會 SITCA | 僅基本資料與連結，無集中持股揭露 |
| **各投信官網** | ✅ 法定每日揭露處（PCF/申購買回清單），HTML 表格或檔案，欄位含代號、名稱、權重、股數 → **持股資料唯一來源，需自建爬蟲** |
| zdsetf.com、etfinfo.tw | 既有第三方同類服務，資料源未公開 → 僅作上線後的數字對照組，不作為資料源 |

**行情資料**：FinMind 為主（可回補歷史、上市上櫃一個 dataset 全包、有含息的加權報酬指數 `TaiwanStockTotalReturnIndex`；每日僅需 2–3 次呼叫，免費層 600 次/時綽綽有餘），TWSE OpenAPI 作備援介面。

> 2026-07-07 實作查證：`TaiwanStockTotalReturnIndex` 以目前 token 可真呼叫，值欄位為 `price`；但 `TaiwanStockPrice` 與 `TaiwanStockPriceAdj` 真呼叫皆回 400，訊息為目前帳號等級不足，且 FinMind 官方文件標示還原股價 dataset 只限 backer/sponsor 會員使用。此處不可改用未還原價替代，因 spec §6 明定所有報酬用還原價；上線前需決定升級 FinMind、改接其他免費價格來源，或自建除權息還原流程。

## 3. 整體架構

```
GitHub Actions（每日排程）
  ├─ 18:30 主流程：抓 27 檔 PCF（15 家投信 adapter）→ 抓 FinMind 行情
  │         → 驗證 → 入庫 → 計算異動/報酬/勝率
  └─ 21:30 補抓場：只重跑失敗/未更新的 ETF
        │
        ▼ 寫入（service key）
  Supabase Postgres（原始快照 + 異動事件 + 指標快取）
        │
        ▼ 唯讀查詢（anon key + RLS）
  Next.js on Vercel（Dashboard，不做計算只做呈現）
```

原則：

- **計算在寫入時做**：異動與指標每日算一次存表，前端只 SELECT
- **原始資料 append-only**：`holdings_snapshot` 為事實來源，任何衍生表可重算
- **爬蟲失敗可見**：記錄於 `scrape_log`，Dashboard 顯示資料缺口警示

技術棧：爬蟲 Python（pandas）、資料庫 Supabase Postgres、前端 Next.js（App Router）+ Tailwind + shadcn/ui + Recharts。

## 4. 資料庫模型

| 表 | 用途 | 關鍵欄位 |
|---|---|---|
| `etf` | ETF 基本檔 | etf_id (PK)、名稱、投信、上市日、PCF 網址 |
| `holdings_snapshot` | 每日持股快照（append-only 事實來源） | PK=(etf_id, trade_date, stock_id)、股數、權重% |
| `holding_change` | 異動事件 | etf_id、trade_date、stock_id、類型(NEW/ADD/TRIM/EXIT)、股數變化、權重變化 |
| `stock_price` | 個股/ETF/指數日收盤（還原價） | stock_id、trade_date、收盤價、還原價 |
| `stock_info` | 代號→名稱→產業別對照 | 來自 FinMind TaiwanStockInfo |
| `etf_metrics` | 每日 ETF 指標快取 | 報酬(1/3/6月、1年、上市以來)、兩種勝率、vs 基準 |
| `scrape_log` | 爬蟲執行記錄 | etf_id、trade_date、狀態、錯誤訊息 |

## 5. 爬蟲層

```
scraper/
├── adapters/          # 一家投信一個 adapter，實作 fetch(etf_id) -> [{stock_id, 股數, 權重}]
├── registry.py        # ETF ↔ adapter 對照；新 ETF 通常只加一行
└── pipeline.py        # 主流程：跑全部 → 驗證 → 入庫 → 記 log
```

- **隔離**：任一 adapter 失敗不影響其他家；失敗標記後由 21:30 場與隔日補抓（回看 3 天）
- **入庫前三道驗證**（任一不過 = 標失敗、不寫入，錯資料比缺資料危險）：
  1. 權重總和落在 70%–101%
  2. 持股筆數 vs 前日無突變（防解析到一半）
  3. 股票代號存在於 `stock_info`
- **異動判定以股數為準**：股數沒變只有權重變 = 純價格波動，不記事件；權重變化 < 0.05pp 忽略。兩個條件須同時成立才記事件——申購/贖回造成的全持股等比例股數變動，權重幾乎不動，會被權重門檻自然過濾，不會產生假異動
- **海外 IP 風險**：實作第一步跑探測腳本確認 15 家官網是否擋 GitHub Actions IP；若擋，備案 (a) 便宜代理 (b) 該幾家改由使用者 Mac 排程跑推上 Supabase
- **禮貌**：每日僅 ~27 次請求，間隔 1–2 秒，正常 User-Agent

## 6. 指標定義

所有價格用**還原價**（除權息調整），基準：ETF 層級 vs **0050（還原）**，個股層級 vs **加權報酬指數**。

**報酬**：近 1/3/6 月、1 年、上市以來（ETF 市價計）。期間不滿者顯示 `—`。旁列同期基準。

**擇時勝率**：完整月份月報酬贏過 0050 的佔比，格式 `67%（8/12 個月）`，永遠帶樣本數。

**選股勝率**：
- 事件 = NEW，或 ADD 且股數增幅 ≥ 10%（過濾申購造成的等比例放大）
- **計分視窗 = 買進日 → (出清日 or 最新交易日)**：
  - 已出清（不論持有幾天）：以實際持有期間結算，不受門檻限制
  - 未出清且滿 **5 個交易日**：計未平倉浮動分，每日更新，出清時自然收斂為定案分（門檻為設定檔常數 `MIN_OPEN_SCORING_DAYS = 5`。查證結論：各 ETF 換股頻率差異極大——單日實測有 ETF 異動 122 筆、也有多檔為 0，且無公開的每檔持有天數統計——故此值定位為務實起始值，未平倉分與已實現分拆開顯示可容忍雜訊；累積三個月自有資料後以實測持有天數分布回測調整）
  - 未出清且未滿 5 個交易日：不計入勝率（防雜訊），但在「新倉追蹤」全揭露
- 部分減持（TRIM）不結束視窗，僅 EXIT 出場
- 顯示拆已實現/未平倉：`61%（已實現 14/22、未平倉 8/14）`；樣本 <10 淡化並標「樣本不足」
- 頁面註明：此為訊號品質指標，未計部位大小，不等於績效貢獻；日內沖銷不會出現在每日 PCF 快照中，無從計分

**操作風格指標**（獨家數據，市面無人提供；由自有累積資料計算）：
- **中位數持有天數**：已出清回合（NEW→EXIT）持有天數的中位數
- **週換手率**：每週持股組合的汰換比例（新進+出清檔數 / 平均持股檔數）
- 用途：量化「誰是快手、誰是長線」，列入排行榜可排序欄位與 ETF 個別頁

## 7. Dashboard 頁面

**① 今日總覽（首頁）**
- 日期選擇器（預設最新交易日）
- 異動牆：NEW/EXIT 置頂，其次 ADD/TRIM，每筆含 ETF、股票、股數與權重變化
- **新倉追蹤雷達**：未滿 20 日建倉部位 + 買進至今超額報酬（每日更新），> ±10% 標色（紅漲綠跌）、標持有天數、標「N 檔 ETF 近期同步建倉」共同訊號；部位滿 20 個交易日「畢業」移出雷達並獲得「長抱」徽章
- **集體動向**：時間切換 `當日/本週/本月`（預設當日），被最多檔 ETF 加碼/減碼 Top 10；排序先 ETF 檔數、後合計權重增量

**② ETF 排行榜**：27 檔總表（規模、各期報酬、兩勝率含樣本數、vs 0050 與大盤、中位數持有天數、週換手率），可排序，贏基準標色。

**③ ETF 個別頁**：績效卡（含操作風格指標）、當前持股表（附 vs 前日 / vs 20 日前權重變化、**持有天數欄**——自最近一次 NEW 起算，滿 20 個交易日標「長抱」徽章）、產業圓餅圖（依 `stock_info` 產業別）、異動時間軸、單一持股權重歷史折線。

**④ 個股反查頁**：誰持有/誰加誰減、全體主動式 ETF 對該股合計權重走勢、事件歷史。

**全站**：資料缺口黃色警示條、手機響應式、深色模式。

## 8. 部署與排程

| 服務 | 角色 | 費用 |
|---|---|---|
| GitHub 公開 repo | 程式碼 + Actions 排程（Secrets 存金鑰） | $0 |
| Supabase | Postgres；RLS 匿名唯讀，寫入僅 service key | $0（500MB） |
| Vercel Hobby | Next.js 部署 | $0（個人非商用） |

- 排程（台北時間）：18:30 主流程、21:30 補抓；非交易日跳過（FinMind `TaiwanStockTradingDate` 判斷）
- 連兩場失敗 → 自動開 GitHub Issue 告警 + Dashboard 黃條
- Supabase 免費層 7 天閒置暫停：每日寫入天然保活
- **上線首日**：一次性回補各 ETF 上市日至今的股價/指數（報酬與擇時勝率即刻可算）；持股明細自上線日起累積（歷史 PCF 無法回補，故越早上線越好）

## 9. 錯誤處理總表

| 情境 | 行為 |
|---|---|
| 單一投信爬取失敗 | 隔離、記 log、21:30 與隔日補抓、Dashboard 黃條 |
| 驗證不過（權重和/筆數/代號） | 視同失敗，不寫入髒資料 |
| FinMind 掛掉 | 切 TWSE 備援介面；當日指標延後一日補算 |
| Actions 整日未跑 | 隔日回看 3 天自動補；PCF 超過揭露窗口即永久缺口，頁面如實標示 |
| 新 ETF 上市 | registry 加一行（同投信）或新增 adapter（新投信） |

## 10. 明確不做（YAGNI）

- 不做使用者帳號/登入（全公開唯讀）
- 不做盤中即時更新（PCF 本來就一天一次）
- 不做自動交易或投資建議（頁面附免責聲明）
- 不爬 zdsetf.com（僅人工對照驗數）
- 不做 D 結尾債券型 ETF（範圍外，架構上 registry 可日後擴充）
